### 2. 从 benchmark 引用转向文档优先引用

- 不再默认先引用 benchmark 题目本身

这样可以避免“拿题目回答题目”，让 RAG 更接近真实资料驱动。

### 3. 第一版文档切片管线

为了让 RAG 真正有“原始资料证据”，又增加了文档切片能力：

- 从 `docx / pdf / text / log` 中抽取 chunk
- 将结果写入 `data/chunks/doc_chunks_v1.jsonl`

后续又继续做了两轮增强：

- 标题感知切片
- 表格感知切片
- 去重和质量过滤

最终把原始粗粒度 chunk 集，收敛成更适合检索的稳定语料。

### 4. citation rerank 与 query-aware rerank

为了让引用更像真实工业证据，而不是索引摘要，又继续加了两层排序优化：

- citation rerank
  - 优先 `WORKLOG-*`
  - 优先 `CASE-*`
  - 优先 `LOG-*`
- query-aware rerank
  - `RPMsg` 问题优先 `DOC-*`
  - 控制链路问题优先日志与工作记录
  - 对 benchmark 模板噪声做惩罚

### 5. 问答质量 benchmark

为了避免 `ask` 只是“看起来能答”，还专门增加了 RAG answer-quality benchmark。

这套评测会检查：

## 七、无硬件版环境打通

### 1. 发现原始阻塞点

在尝试真实执行 `SCRIPT-004` 时，首先遇到的是：

- WSL 可用
- `python3` 可用
- 但脚本依赖 `/home/librobot.so.1.0.0`

进一步检查发现：

- 仓库资料里确实有 `librobot.so.1.0.0`
- 但它是 `ARM aarch64` 版本
- 当前 WSL 是 `x86_64`

也就是说，就算把库拷进去也无法直接在当前环境加载。

### 2. 尝试编译真实 stub `.so`

中途评估过一条方案：

- 在 WSL 内编译一个 x86_64 版 stub `librobot.so`

但继续探查后发现：

- 当前 WSL 没有编译器
- 需要 `sudo apt-get install build-essential`
- 这台环境并不能稳定走无交互 `sudo`

因此这条方案不够稳，不适合当下“离线无硬件打通”的主要路径。

### 3. 切换到 wrapper + Python stub 方案

最终选择了一条更稳、侵入更小的方案：

- 不修改原始 probe 脚本
- 新增一个 WSL Python wrapper
- 在运行时拦截 `ctypes.CDLL("/home/librobot.so.1.0.0")`
- 用纯 Python 的假总线实现替代真实共享库

新增文件：

- `scripts/wsl/run_with_stub.py`
- `scripts/wsl/librobot_stub_runtime.py`
- `scripts/setup_wsl_stub.ps1`
- `docs/wsl_stub_mode.md`

同时在 CLI 中新增：

- `ieda tools doctor`
- `ieda tools setup-stub`

### 4. 无硬件模式的效果

这套 stub 模式启用后，即使不存在真实 `/home/librobot.so.1.0.0`，也可以运行：

```powershell
ieda tools run "运行 tmp_probe_can_heartbeat.py 采集 8 轮状态快照，然后把结果汇总给我。" --execute
```

实测结果：

- `returncode = 0`
- `parsed_output.status = ok`
- `poll_count = 8`
- `axis0.statusCode = 33`
- `axis1.statusCode = 33`
- `axis0.encoder = 1008`
- `axis1.encoder = 2008`

说明：

- 工具层确实发起了真实执行
- 原始探针脚本确实跑起来了
- 结果解析器确实拿到了结构化输出
- 无硬件环境下也能把执行链调通

这一点对后续非常重要，因为它让项目在离线场景下也能持续开发，而不是必须等到回公司接真机才能推进。
